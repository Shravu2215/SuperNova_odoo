AssetFlow — Backend & API PRD
Scope: Backend Engineer's Track (Node.js + Express +
TypeScript + Prisma)
Context: This is a team hackathon build. Database schema is owned by
the DB teammate (Prisma schema already exists / is being ﬁnalized in
parallel). Frontend is owned by the frontend teammate. This document
scopes ONLY the backend service: APIs, business logic, auth, RBAC,
workﬂows, validation, security, real-time events, and testing. Written
so Claude Code can implement directly with minimal ambiguity.
Assumption: The Prisma schema described in §1 (Data Contract) is the
agreed contract with the DB teammate. If the actual schema differs,
treat §1 as the source of truth to build against and ﬂag drift — do not
silently redesign it.

2. Backend Deliverable Deﬁnition
A standalone Express + TypeScript API service that:
1. Exposes a fully working REST API per §6, matching the response
envelope in §5.
2. Owns all business rules, state machines, and validation (frontend
must not be trusted for any of this).
3. Emits Socket.IO events for real-time consumers (dashboard,
notiﬁcations).
4. Is independently testable via a Postman collection, without
requiring the frontend to exist.
5. Runs standalone via npm run dev  against a DATABASE_URL  env
var, and deploys to Render via npm run build && npm start .
3. Tech Stack (backend slice)
Node.js + Express.js + TypeScript
Prisma ORM (client only — schema owned by DB teammate)
JWT (jsonwebtoken ) + bcrypt for auth
Zod for request validation
Helmet, express-rate-limit, CORS
Socket.IO server
Postman for endpoint testing (export collection to /postman )
dotenv for conﬁg; no secrets hardcoded

5. API Response Contract (must be identical
across all endpoints)
// success
{ success: true, data: T, message?: string }
// failure
{ success: false, error: { code: string, message: string,  
details?: unknown } }
HTTP status codes must match semantics: 200 OK, 201 Created,
400 validation, 401 unauthenticated, 403 forbidden
(role/ownership), 404 not found, 409 conﬂict (double allocation,
booking overlap, duplicate email), 500 only for truly unexpected
errors (never leak stack traces — log server-side via logger.ts ,
return a generic message to the client).
Pagination convention for list endpoints: ?page=1&limit=20 ,
response data: { items: T[], total, page, limit } .

6.2 Departments (/api/departments) — Admin only for writes
GET / — list with hierarchy (nested or ﬂat + parentId, frontend
decides rendering).
POST / — create (name, parentId?).
PATCH /:id  — edit / toggle isActive .
Validation: prevent circular parent references (a department cannot
be its own ancestor) — this is backend's job to catch.
6.3 Categories (/api/categories) — Admin only for writes
CRUD + customFields  as a schema deﬁnition (array of { key,
label, type } ) stored in Json.
On asset creation, validate submitted customFields  values
against the category's ﬁeld deﬁnitions (type-check at minimum).
6.4 Employees (/api/employees) — Admin only
GET / — list/search/ﬁlter by department, role, active status,
paginated.
PATCH /:id/role  — the only endpoint in the entire system that
can change role. Must:
Reject if actor is not ADMIN.
Reject self-demotion of the last remaining Admin (prevent
lockout).
Write an ActivityLog  entry with old role →  new role.
Emit a Notification  to the affected user.
6.5 Assets (/api/assets)
POST / (Asset Manager/Admin) — register asset:
Generate assetTag  sequentially and atomically (use a DB
transaction with SELECT ... FOR UPDATE  on a counter row, or
a Postgres sequence — neverCOUNT(*) + 1 , it races under
concurrent writes).
Write AssetHistory  entry "REGISTERED" .
GET / — list with ﬁlters (status, category, department of current
holder, search by tag/name/serial), paginated.
GET /:id  — full detail.
GET /:id/history  — timeline of AssetHistory , newest ﬁrst.
PATCH /:id  (Asset Manager/Admin) — edit metadata (not
status — status only changes via workﬂow endpoints below).
DELETE /:id  (Admin) — soft delete (isDeleted = true ), only if
not currently ALLOCATED .
6.6 Allocation Workﬂow (/api/allocation-requests)
Core differentiator module — build carefully.
POST / — body: { assetId, type:
ALLOCATION|TRANSFER|RETURN, targetUserId?, reason? }
Allocation: inside a DB transaction, re-check Asset.status ===
AVAILABLE . If not, return 409 with message "Asset is
already allocated to {holderName}" . If available, either
auto-approve (if requester is Manager/Admin) or create
PENDING  request for manager approval — conﬁrm which ﬂow
with team, default to requires manager approval for MVP
consistency.
Transfer: requires an existing holder; creates PENDING  request;
only ASSET_MANAGER /ADMIN can approve.
Return: holder or manager can initiate; on approval, asset
condition is recorded, status →  AVAILABLE  (or
UNDER_MAINTENANCE  if condition marked DAMAGED ).
PATCH /:id/approve  (Manager/Admin) — transactionally: update
Asset.status  + currentHolderId , write AssetHistory , mark
request APPROVED , emit Notification  to requester + previous
holder (if transfer), emit socket event asset:updated .
PATCH /:id/reject  (Manager/Admin) — mark REJECTED  with
reason, notify requester.
GET / — list, ﬁlterable by status/type/asset/user, paginated.
Concurrency requirement: the availability check + status update must
happen inside a single Prisma $transaction  to prevent race
conditions from two simultaneous approval calls.
6.7 Bookings (/api/bookings)
GET /rooms  — list active meeting rooms.
POST /bookings  — body: { roomId, startTime, endTime,
purpose? }
Overlap check (in transaction): reject if any existing BOOKED
booking for the same room satisﬁes existing.startTime <
newEnd AND existing.endTime > newStart . Return 409 with
the conﬂicting slot info.
On success, emit socket event booking:created  (so other
clients viewing that room's calendar update live).
DELETE /bookings/:id  — owner or Admin only; sets status =
CANCELLED  (don't hard-delete, preserves history).
GET /bookings?roomId=&date=  — for calendar view.
6.8 Maintenance (/api/maintenance)
POST / — any authenticated user raises a ticket on an asset they
currently hold (or Manager on any asset). Status starts REQUESTED .
PATCH /:id/approve  (Manager/Admin) — status →
APPROVED /IN_PROGRESS , Asset.status →  UNDER_MAINTENANCE ,
write AssetHistory .
PATCH /:id/reject  (Manager/Admin) — status →  REJECTED ,
notify raiser with reason.
PATCH /:id/complete  (Manager/Admin) — status →
COMPLETED , Asset.status →  AVAILABLE , write AssetHistory .
GET / — list/ﬁlter by status/asset, paginated.
6.9 Audits (/api/audits)
POST / (Manager/Admin) — start an AuditRecord , auto-populate
AuditItem  rows for every non-deleted asset in scope (all, or
ﬁltered by category/department if a scope param is passed).
PATCH /:auditId/items/:itemId  — record FOUND | MISSING |
DAMAGED  + notes.
POST /:auditId/complete  — set completedAt ; for every item
marked MISSING , transactionally update the linked Asset.status
→ LOST and write AssetHistory ; generate a discrepancy
summary in the response ({ found, missing, damaged, total
}).
GET /:auditId  — full report.
6.10 Reports (/api/reports) — Manager/Admin
GET /most-used  — assets ranked by allocation count/duration
(aggregate query over AssetHistory /AllocationRequest ).
GET /least-used  — inverse.
GET /department-usage  — asset counts grouped by current
holder's department.
GET /maintenance-forecast  — assets with upcoming/recurring
maintenance patterns (simple heuristic acceptable for MVP: assets
with 2+ maintenance tickets in last N days).
All report endpoints should be pure read/aggregate — no side
effects, cache-friendly.
6.11 Notiﬁcations (/api/notifications)
GET / — current user's notiﬁcations, paginated, ?
unreadOnly=true  ﬁlter.
PATCH /:id/read  — mark single read.
PATCH /read-all  — mark all read.
Notiﬁcations are created server-side only, as a side effect of other
module actions (never a public "create notiﬁcation" endpoint) —
implement via a shared notificationService.notify(userId,
type, message)  helper called from other services.
6.12 Activity Logs (/api/activity-logs) — Admin only
GET / — ﬁlterable by entityType , entityId , userId, date
range, paginated.
Writes happen via a shared activityLogger.log(...)  helper
invoked from every state-changing service — do not rely on a global
middleware to infer this generically; be explicit per action so
action/entityType  values stay meaningful (e.g.
"ASSET_ALLOCATED" , "ROLE_CHANGED" , "BOOKING_CANCELLED" ).
6.13 Dashboard (/api/dashboard/summary)
Role-aware aggregate: Employee gets counts scoped to self;
Manager/Admin gets org-wide.
Returns: availableAssets, allocatedAssets,
maintenanceToday, roomsBookedToday, overdueReturns .
Should be a single eﬃcient query set (use Promise.all  of count
queries, not N+1).
7. Business Rules & State Machines (backend-
enforced, non-negotiable)
Role elevation: only via PATCH /employees/:id/role , only by ADMIN.
Every other endpoint must strip/ignore any role ﬁeld in the request
body.
Asset status transitions — implement as a single guarded function, e.g.
assetService.transitionStatus(assetId, newStatus, context) ,
called from every module that changes asset state. Reject invalid
transitions with 400:
AVAILABLE →  ALLOCATED | UNDER_MAINTENANCE | LOST |  
RETIRED
ALLOCATED →  AVAILABLE (via return/transfer completion)
UNDER_MAINTENANCE →  AVAILABLE
RETIRED →  DISPOSED
LOST → (terminal, requires manual Admin override if ever  
recovered — separate explicit endpoint, not part of MVP  
unless time allows)
No double allocation: enforced via transaction + status check, not via
unique constraints alone (a currentHolderId  unique-per-asset is
implicit in the model, but the check-then-act must be atomic).
No booking overlap: enforced via transaction + overlap query, per room.
Approval workﬂows always follow: requested →  (approved |
rejected) , and any status-changing approval must be wrapped in a
Prisma $transaction  covering: the request status update + the entity
status update + the AssetHistory /audit trail write + notiﬁcation
creation. All-or-nothing.

9. Validation Strategy (Zod)
One schema ﬁle per module (*.schema.ts ), exporting
createXSchema , updateXSchema , queryXSchema .
validate.middleware.ts  takes a schema + target
(body/query/params) and rejects with 400 + ﬁeld-level errors
on failure, before the controller runs.
Reject unknown/extra ﬁelds (.strict()  on Zod objects) — this is
what prevents a client from sneaking a role ﬁeld into a signup
payload.
10. Security Checklist (backend-owned)
[ ] Helmet applied globally in app.ts.
[ ] CORS locked to the frontend's deployed origin + localhost dev
origin via env var.
[ ] express-rate-limit : strict on /api/auth/*  (e.g. 10 req/min),
moderate globally (e.g. 100 req/min).
[ ] JWT secrets, DB URL, and all conﬁg pulled from .env, validated
at boot via config/env.ts  (fail fast, don't start server with
missing/invalid env).
[ ] toSafeUser()  (or Prisma select) used everywhere a User is
returned — never leak passwordHash .
[ ] Centralized error.middleware.ts  — catches all thrown errors,
logs full detail server-side, returns sanitized message to client,
never a raw stack trace.
[ ] RBAC middleware applied per-route, not assumed from frontend
hiding UI elements.
[ ] All list endpoints paginated (no unbounded findMany ).
[ ] Prisma used exclusively — no raw SQL string interpolation.
